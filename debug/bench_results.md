# Qwen3-235B-A22B 并行策略基准 (8×H200, dummy 权重)

测量方式：单请求（并发=1），客户端流式时间戳测 TTFT / prefill / decode，**不依赖服务端 usage（其字段恒为 0）**。
每请求加唯一随机前缀防 radix 缓存命中；首请求预热丢弃。

## PP=8 (`--pp-size 8 --num-pages 200000`)

| 输入 | TTFT (s) | prefill tok/s | decode tok/s | 显存/卡 (MiB) |
|---|---|---|---|---|
| 8k   | 1.33  | 6,026 | 50.3 | 66,167 |
| 32k  | 7.15  | 4,476 | 47.5 | 66,681 |
| 128k | 60.4  | 2,119 | 21.8 | 66,681 |

> PP 显存低是因为用了 `--num-pages 200000`（小 KV 池），与 TP/EP 的 0.9 ratio 不可直接比较。
> PP 单请求超过 128k 极慢（stage 串行无重叠），未测 256k/512k。

## 结论（单请求 / 低并发场景）

- **TP=8 最优**：prefill 和 decode 都最快，单请求把整个模型并行到 8 卡。长上下文单请求服务首选。
- **EP=8**：prefill 与 TP 接近（略低）；**单请求 decode 严重变差**（8 vs TP 77 tok/s）——每个 MoE 层每 token 要做多轮 all-to-all + all-gather，通信延迟主导。EP 的收益在**大批量 MoE decode**（摊薄 all-to-all），不是单请求。
- **PP=8**：单请求 **prefill 最慢**（stage 串行 + P2P，无并行）；decode 居中。PP 的收益在**高并发吞吐**（流水线填满）与按层切分省显存，不是单请求延迟。

## 引擎稳定性边界（真实发现）

- **pynccl 自定义核在 >256k 上下文崩溃**（`CUDA illegal memory access`，异步报在 pynccl all-reduce）。必须用 `--disable-pynccl`（标准 NCCL）才能到 512k。
- **~1M 上下文崩溃**：很可能是 attention 索引 int32 溢出（1M×32heads×128 ≈ 4e9 > 2^31）。单请求可行上限 ≈ **512k**（2M 在 8×H200 也放不下，KV 需 ~770GB）。

## 指标坑（已规避）

1. **radix 前缀缓存命中**：重复文本 prompt + 复用服务器 → 短 prompt 命中长 prompt 缓存 → prefill 假高（37 万 tok/s）。修复：每请求唯一随机前缀。
2. **固定 rng 种子跨运行命中**：同种子在复用服务器上仍命中上次缓存。修复：按运行唯一种子。
3. **首请求 warmup**：第一个测量点被一次性初始化开销污染。修复：预热请求丢弃。
4. **服务端 usage 全 0**：完全客户端测量。
